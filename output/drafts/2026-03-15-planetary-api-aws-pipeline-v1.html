Building a Secure AWS CI/CD Pipeline for a Deliberately Vulnerable Flask API

How I wired up Semgrep, Snyk, Postman, and AWS Security Hub into a CodePipeline that detects vulnerabilities before they reach production — even when the vulnerabilities are intentional.

I recently built planetary-api as a security training project. It's a Flask API for Star Trek planetary data that deliberately includes vulnerable endpoints — SQL injection, command injection, path traversal, and SSRF — sitting right alongside secure versions of the same endpoints. The point is to show the contrast.

But having vulnerable code doesn't mean you want a chaotic pipeline. I wanted to build a proper AWS CodePipeline around it — one that detects those vulnerabilities, reports them centrally, and blocks the pipeline unless someone consciously decides to let it through. This post walks through how that works.

The infrastructure — three CDK stacks

Everything is provisioned with AWS CDK (Cloud Development Kit) in Python. Three stacks do the work:

  • PlanetaryEcr — an ECR repository for the Docker image
  • PlanetaryEcs — VPC, ECS Fargate cluster, Application Load Balancer (ALB), and RDS MySQL 8.0
  • PlanetaryPipeline — the CodePipeline itself, all CodeBuild projects, S3 artifact store, SNS, SSM parameters, and IAM roles

Deploying is a three-step process: deploy ECR first, then ECS, then the pipeline stack. The pipeline triggers automatically on any push to the master branch via a CodeStar GitHub connection.

Stage 1 and 2 — source and build

Stage 1 is the GitHub source connection via CodeStar. Stage 2 builds the Docker image, tags it with the commit SHA, and pushes to ECR.

phases:
  pre_build:
    commands:
      - echo Logging into Docker Hub...
      - echo $DOCKERHUB_PASSWORD | docker login -u $DOCKERHUB_USERNAME --password-stdin
      - aws ecr get-login-password --region $AWS_DEFAULT_REGION | docker login --username AWS --password-stdin $ECR_REPO
  build:
    commands:
      - docker build -t $IMAGE_REPO_NAME:$COMMIT_SHA .
      - docker tag $IMAGE_REPO_NAME:$COMMIT_SHA $ECR_REPO:$COMMIT_SHA
      - docker tag $IMAGE_REPO_NAME:$COMMIT_SHA $ECR_REPO:latest
  post_build:
    commands:
      - docker push $ECR_REPO:$COMMIT_SHA
      - docker push $ECR_REPO:latest
      - printf '[{"name":"%s","imageUri":"%s"}]' "$CONTAINER_NAME" "$ECR_REPO:$COMMIT_SHA" > imagedefinitions.json

I log into Docker Hub before authenticating to ECR — without credentials, CodeBuild's shared IPs hit Docker Hub rate limits fast. Both sets of credentials come from Secrets Manager, not hardcoded into the buildspec.

Stage 3 — three parallel security scans

Stage 3 runs three CodeBuild actions in parallel, each targeting a different type of vulnerability.

Semgrep SAST

Semgrep reads the source code and flags patterns that match known vulnerability types. I run it with --config=auto to pull in Semgrep's maintained ruleset, outputting SARIF (Static Analysis Results Interchange Format).

semgrep scan --config=auto --sarif --output=semgrep.sarif .

Security Hub expects findings in ASFF (Amazon Security Finding Format), not SARIF. I wrote a small conversion script, sarif_to_asff.py, that maps the fields across and batch-imports the results. The raw SARIF and ASFF files also land in S3.

Semgrep also runs as a GitHub Actions check on pull requests, so you get static analysis feedback before anything reaches the pipeline.

Snyk SCA

Software Composition Analysis (SCA) scans your dependencies, not your code. Snyk checks requirements.txt against its vulnerability database and flags anything with a known CVE. I also generate a Software Bill of Materials (SBOM) in CycloneDX format alongside the scan:

snyk test --file=requirements.txt --json > snyk_results.json || true
cyclonedx-bom -r -o bom.xml

Same pattern as Semgrep: snyk_to_asff.py converts the JSON output to ASFF and imports it to Security Hub. The SBOM, raw results, and ASFF all go to S3.

Postman security tests

Dynamic testing means actually firing requests at a running instance of the API. This runs the "Security" collection in Newman — test cases for SQL injection payloads, command injection strings, and XSS attempts.

The buildspec spins up MySQL and the API container using Docker-in-Docker, waits for them to be healthy, then runs Newman:

docker-compose up -d
sleep 20
newman run postman/planetary-api.postman_collection.json \
  --folder Security \
  --environment postman/local.postman_environment.json \
  --reporters cli,junit,html \
  --reporter-junit-export newman-results.xml \
  --reporter-html-export newman-report.html \
  --suppress-exit-code

The --suppress-exit-code flag is intentional. I don't want a failed Postman test to kill this stage — the results feed into Security Hub, and the gate in Stage 4 handles the blocking decision. Separating reporting from gating keeps both cleaner.

Stage 4 — the security gate

After all three scans complete, the security gate queries Security Hub for active HIGH or CRITICAL findings from the Semgrep and Snyk generators:

aws securityhub get-findings \
  --filters '{
    "GeneratorId": [
      {"Value": "planetary-api-semgrep", "Comparison": "CONTAINS"},
      {"Value": "planetary-api-snyk", "Comparison": "CONTAINS"}
    ],
    "SeverityLabel": [
      {"Value": "HIGH", "Comparison": "EQUALS"},
      {"Value": "CRITICAL", "Comparison": "EQUALS"}
    ],
    "WorkflowStatus": [{"Value": "NEW", "Comparison": "EQUALS"}],
    "RecordState": [{"Value": "ACTIVE", "Comparison": "EQUALS"}]
  }'

If findings exist, it checks an SSM Parameter Store value: /planetary-api/pipeline/security-override. That defaults to "false". If it's false and there are findings, the buildspec exits with code 1 and the pipeline stops.

The override mechanism: Because this is a training project with intentional vulnerabilities, I need a way to say "yes, I know about these findings, let it through anyway." Setting the SSM parameter to "true" allows the pipeline to proceed past the gate. After each pipeline run, the parameter resets back to "false" automatically.

The gate doesn't stop the vulnerable code from existing — it's intentionally there. What it does is force a deliberate call. Nobody ships a high-severity finding by accident.

Stage 5 — manual approval

Even when the security gate passes, a human signs off before anything hits production. SNS sends an email with a direct link to Security Hub so the approver can see the current findings before clicking approve.

Stages 6–10 — smoke test, deploy, migrate, lockdown, verify

Stage 6 runs functional smoke tests with Newman before touching production — a "does this thing actually work?" check using the "Basic" and "Negative" Postman test folders.

Stage 7 does the ECS Fargate rolling update, passing the imagedefinitions.json from Stage 2 so ECS knows which image to pull.

Stage 8 runs a one-off ECS task to run flask db_create && flask db_seed against the production RDS. Both commands are idempotent, so running them on every deploy is safe — they won't wipe data that's already there.

Stage 9 — Lockdown — optionally restricts the ALB security group to a specific CIDR. If you set the AllowedIp pipeline variable, the buildspec revokes all existing port-80 rules and replaces them with that CIDR. If you leave it blank, it stays open.

Stage 10 waits 15 seconds for ECS to stabilise, then does a live curl health check against the ALB DNS name to confirm the deployment worked.

Secrets management

Nothing is hardcoded. Docker Hub credentials, Snyk token, Semgrep token, database password — all come from AWS Secrets Manager at runtime via the buildspec's env.secrets-manager block. The S3 bucket is versioned, encrypted, and blocks all public access.

env:
  secrets-manager:
    DOCKERHUB_USERNAME: "planetary-api/dockerhub:username"
    DOCKERHUB_PASSWORD: "planetary-api/dockerhub:password"
    SNYK_TOKEN: "planetary-api/snyk:token"

Local dev hooks

The pipeline isn't the first line of defence. Talisman runs as a pre-push hook and scans for secrets before code even reaches GitHub. Black and Flake8 catch formatting issues as a pre-commit hook. And Semgrep runs again in GitHub Actions on pull requests, so you're seeing SAST results at PR review time, not after a full pipeline run.

Wrapping up

The layered approach is what I'm happiest with here. Local hooks, GitHub Actions SAST on PRs, and then the full pipeline scan before anything ships — three chances to catch something before it matters.

The intentional vulnerabilities actually make this a better demo than a clean codebase would. The pipeline flags them, Security Hub logs them, and the gate blocks on them. You have to make an active decision to override. That feels like the right behaviour for a security pipeline: not quietly ignoring risk, but making it visible and forcing a conscious choice.

CDK stacks and buildspecs are all at github.com/andyrat33/planetary-api.
